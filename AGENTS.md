# 오밥완(obabwan) 프로젝트 운영 지침

이 문서는 이 저장소에서 개발, 수정, 검토 작업을 수행할 때 항상 준수해야 하는 프로젝트 지침이다.

작업을 시작할 때 현재 구현 상태와 최근 결정 사항이 정리된 `PROJECT_CONTEXT.md`와 전체 작업 이력인 `누적로그.txt`를 먼저 읽는다. 기능이나 운영 방식이 바뀌면 컨텍스트 문서는 짧고 최신 상태로 갱신하고, 누적 로그에는 기존 내용을 지우지 말고 날짜·변경·검증·커밋을 이어서 기록한다.

## 1. 아키텍처 및 기술 스택

- 애플리케이션은 단일 `index.html` 파일을 중심으로 동작하는 제로 빌드 웹앱으로 유지한다.
- Vanilla JavaScript, Tailwind CSS, HTML5 Canvas, Web Audio API를 사용한다.
- 외부 라이브러리는 최소화하고, 필요한 경우 CDN으로 불러온다.
- GitHub Pages에 배포한다: <https://dmazone.github.io/obabwan/>
- 모든 로컬 리소스 참조는 `./`로 시작하는 상대 경로를 사용한다.
- 별도의 유료 서버를 두지 않는 0원 인프라 및 서버리스 구성을 우선한다.

## 2. 수익화 파이프라인

- 광고 채널은 텐핑, 카카오 애드핏, 쿠팡 파트너스를 고려한다.
- 광고 스크립트와 광고 실행 영역은 샌드박스로 격리한다.
- 상품 구매 전환은 쿠팡 파트너스 딥링크 매핑을 통해 처리한다.
- 처방전 생성 기능은 Google Gemini API의 Flash 모델을 사용한다.

## 3. 안정성 및 유지보수

- 모든 URL 파라미터(`title`, `items` 등)는 입력 시 인코딩하고 사용 시 안전하게 디코딩한다.
- URL 파라미터에는 특수문자 스크리닝 및 유효성 검사를 반드시 적용한다.
- 사용자 입력을 DOM에 반영할 때 XSS가 발생하지 않도록 `innerHTML` 사용을 피하고 안전한 텍스트 삽입 방식을 사용한다.
- 전체 화면 높이는 모바일 브라우저 UI를 고려해 `h-[100dvh]`를 사용한다.
- 기기 및 화면 크기 변화에 대응하도록 Offscreen Canvas 기반 리베이킹 시스템을 적용한다.
- 기능 실행 및 종료 시 생성한 전역 객체, 광고 스크립트 참조, 이벤트 리스너, 타이머, 오디오 및 Canvas 관련 리소스를 정리한다.
- 더 이상 필요하지 않은 전역 참조는 `null`로 설정하고, 삭제 가능한 동적 속성은 `delete`로 제거한다.

## 4. 변경 원칙

- 제로 빌드와 단일 HTML 구조를 깨는 프레임워크, 번들러, 백엔드 도입은 명시적인 승인 없이 진행하지 않는다.
- 새 리소스와 링크를 추가할 때 GitHub Pages의 하위 경로 배포 환경에서 정상 동작하는지 확인한다.
- 광고, 제휴 링크, Gemini API 연동 변경 시 실패 격리와 정리 로직을 함께 검증한다.

## 5. 변경 완료 및 상시 배포 절차

- 이 프로젝트의 기능, UI, 문서 또는 설정을 변경한 작업은 로컬 파일 수정만으로 완료 처리하지 않는다.
- 변경 후 `index.html`의 인라인 JavaScript 문법 검사와 로컬 HTTP 응답 검사를 수행한다.
- 검증이 통과하면 변경 파일을 Git에 커밋하고 `origin`의 `main` 브랜치로 푸시한다.
- 커밋 메시지는 변경 목적을 짧고 구체적으로 기록한다.
- 푸시 후 GitHub Pages 배포 상태를 확인하고 <https://dmazone.github.io/obabwan/>에서 최신 화면과 핵심 기능이 동작하는지 검증한다.
- 배포 지연 중에는 완료로 보고하지 않고, Pages 배포가 성공하거나 명확한 실패 원인이 확인될 때까지 상태를 확인한다.
- API 키, 광고 비밀값, 인증 토큰은 정적 HTML이나 Git 이력에 커밋하지 않는다.
- 사용자가 명시적으로 배포 제외를 요청하거나 배포 권한·네트워크 장애가 있는 경우에만 푸시를 생략하고 그 사유를 보고한다.
